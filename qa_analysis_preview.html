QA 판정

현재 상태로는 실제 교회 운영 배포에 부적합합니다.

UI 가림 같은 표면 결함도 있지만, 더 큰 문제는 교회별 데이터 분리, 항목 조인, 동명이인 식별, GET 기반 상태변경입니다. 이 문제들은 화면을 보기 좋게 바꿔도 보고서 금액과 개인별 내역이 계속 틀릴 수 있습니다.

먼저 수정할 축

  • 모든 조회/집계에 church_id 적용
  • account_items 조인을 church_id + target_year + item_type + item_code 기준으로 통일
  • 개인 필터는 이름이 아니라 member_id 기준
  • 모바일 fixed bottom UI 재설계
  • 상태 변경은 POST + CSRF + 감사 로그

문제 설명

지출 화면은 지출 항목별 금액/대상명을 한 번에 넣는 구조라 모바일에서 저장 버튼 접근성은 중요합니다. 그러나 fixed submit 버튼과 fixed bottom nav가 따로 존재하면 키보드가 올라온 상태에서 입력 중인 필드, 저장 버튼, 메뉴가 서로 겹칠 수 있습니다.

수정 방향

  • 모바일에서는 submit과 nav를 하나의 footer action 영역으로 통합
  • 키보드 focus 시 저장 버튼을 inline sticky로 전환
  • safe-area-inset-bottom 반영
  • 긴 항목명은 2줄 line clamp 및 title/상세 표시

설정 화면 오류 재현

수입 항목 저장

INSERT: church_id, target_year, item_code, item_name

item_type 누락 → 목록 SELECT item_type='INCOME'에서 안 보일 수 있음

수입 항목 삭제

DELETE: church_id, target_year, item_code

item_type 조건 없음 → 같은 코드의 지출 항목도 삭제 위험

사용 중 항목 삭제

기록이 있는 항목을 삭제하면 과거 보고서의 항목명이 깨질 수 있으므로 삭제 대신 비활성화가 안전합니다.

권장 설정 플로우

  1. 설정 홈: 수입 항목 / 지출 항목 / 시즌 / 카테고리 분리
  2. 항목 저장: item_type 명시, prepared statement
  3. 항목 삭제: 참조 건수 확인 후 inactive 처리
  4. 긴 항목명/특수문자: 화면과 JS 모두 escape
UNIQUE(church_id, target_year, item_type, item_code) ALTER TABLE account_items ADD COLUMN is_active TINYINT DEFAULT 1;
